# T1059.001 / T1218 — LOLBin Execution via Windows Security Event 4688

**Detection #5 — LOLBin Abuse and Encoded Commands via Process Creation Auditing**

---

## Summary

This case study documents the detection of Living-off-the-Land Binary (LOLBin) abuse and PowerShell encoded command execution using Windows Security Event ID 4688 (process creation) ingested into Splunk. The detection identifies processes with command lines matching known-malicious patterns — inline script execution via `mshta.exe`, `rundll32.exe` javascript/VBScript invocations, and Base64-encoded PowerShell payloads — across multiple MITRE ATT&CK (sub-)techniques in a single query.

| Field | Value |
|---|---|
| **MITRE ATT&CK** | [T1059.001 — PowerShell](https://attack.mitre.org/techniques/T1059/001/), [T1218.005 — Mshta](https://attack.mitre.org/techniques/T1218/005/), [T1218.011 — Rundll32](https://attack.mitre.org/techniques/T1218/011/) |
| **Related** | T1105 — Ingress Tool Transfer, T1140 — Deobfuscate/Decode, T1197 — BITS Jobs |
| **Data source** | Windows Security Event ID 4688 (process creation) |
| **Log index** | `winlogs` |
| **Sourcetype** | `WinEventLog:Security` |
| **Detection type** | Signature / command line pattern matching |

---

## Why this matters

LOLBins are legitimate Windows binaries that attackers abuse to execute malicious code while blending into normal system activity. Because they are signed Microsoft binaries, they bypass application allowlisting, evade many AV signatures, and appear routine to defenders who don't inspect command line arguments. The specific binaries targeted here represent the most commonly observed LOLBin TTPs in real intrusions:

- **`rundll32.exe` with `javascript:`** — executes arbitrary JavaScript through the `mshtml` COM object; no file written, no script interpreter spawned
- **`mshta.exe` with `vbscript:` / `javascript:`** — executes inline VBScript or JavaScript; frequently used for fileless initial access from phishing lures
- **`certutil.exe` with `-decode` / `-urlcache`** — decodes Base64 payloads or downloads files from the internet; commonly used as a downloader in post-exploitation
- **`PowerShell -EncodedCommand`** — Base64-encodes arbitrary PowerShell to obscure payload content from log inspection and basic signature detection

Event 4688's `Process Command Line` field is what makes these detectable — the binary name alone is insufficient, but the full command line exposes the malicious invocation pattern regardless of how the binary was renamed or relocated.

This detection also demonstrates **dual-source coverage** alongside Detection #1 (T1059.001 via Sysmon Event 1): the same encoded PowerShell technique is caught independently by both the Sysmon pipeline and the Windows Security log, validating that neither single source is a single point of failure.

## Lab gotchas: two issues that had to be fixed before Event 4688 was useful

### Issue 1 — Wrong sourcetype in Splunk queries

All previous detections in this lab use `sourcetype=xmlwineventlog` because Sysmon events are collected via the Splunk Add-on for Microsoft Sysmon and normalized into that sourcetype. Windows Security log events collected by the standard Splunk universal forwarder come in under a different sourcetype entirely.

**Symptom:** `index=winlogs sourcetype=xmlwineventlog EventCode=4688` returned zero results despite Event Viewer confirming 4688 events were being generated.

**Diagnosis:** Running a broad field inspection query:

```spl
index=winlogs EventCode=4688 earliest=-24h
| head 1
| table sourcetype, source, *
```

Returned `sourcetype=WinEventLog:Security`, `source=WinEventLog:Security` — not `xmlwineventlog`.

**Fix:** All 4688 queries must use `sourcetype="WinEventLog:Security"`. The two sourcetypes coexist in the same `winlogs` index but are collected and normalized by different mechanisms.

### Issue 2 — Process Command Line field was blank

Despite enabling the audit policy, the `Process Command Line` field in 4688 events was empty — which removes most of the detection value entirely.

**Symptom:** 4688 events appearing in Splunk but `Process_Command_Line` blank for every event.

**Diagnosis:** The registry key controlling command line inclusion was either not set or not applied:

```powershell
reg query "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\Audit" /v ProcessCreationIncludeCmdLine_Enabled
```

**Fix:**

```powershell
reg add "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\Audit" /v ProcessCreationIncludeCmdLine_Enabled /t REG_DWORD /d 1 /f
gpupdate /force
```

Verified by checking that subsequent 4688 events contained populated `Process Command Line` fields in Event Viewer.

### Issue 3 — LOLBin simulation commands error from PowerShell

Running LOLBin simulation commands directly from a PowerShell prompt caused PowerShell to try to parse the command syntax itself rather than passing it verbatim to the target binary.

**Symptom:** `rundll32.exe javascript:...` from PowerShell threw `CommandNotFoundException: alert` because PowerShell intercepted the argument.

**Fix:** Run LOLBin simulation commands from `cmd.exe`, not PowerShell. The process launch — and therefore the 4688 event — occurs regardless of whether the payload executes successfully. The detection captures the command line at process creation time, before any execution error.

### Lessons

- **Sourcetype is not universal.** Different Splunk inputs produce different sourcetypes even when events land in the same index. Always verify with a broad field inspection before writing detection queries.
- **Command line logging is opt-in and easy to miss.** The audit policy enables the event; the registry key separately enables the command line field. Both are required. Neither enables the other.
- **Detection validity doesn't require payload success.** Event 4688 fires at process creation. A LOLBin invocation that errors at runtime still produces a fully populated detection event.

## Simulation

Four simulation commands were run to generate detection events covering three MITRE techniques:

```powershell
# T1059.001 — PowerShell encoded command
$enc = [Convert]::ToBase64String([Text.Encoding]::Unicode.GetBytes('Write-Host "4688 test"'))
powershell.exe -EncodedCommand $enc
```

```cmd
:: T1218.011 — Rundll32 javascript execution (run from cmd.exe)
rundll32.exe javascript:"\..\mshtml,RunHTMLApplication ";alert('lolbin-rundll32')

:: T1218.005 — Mshta VBScript inline execution (run from cmd.exe)
mshta.exe vbscript:Execute("CreateObject(""Wscript.Shell"").Run ""cmd /c echo lolbin-test"":close")
```

```powershell
# T1105/T1140 — Certutil decode (common downloader TTP)
certutil.exe -decode C:\Windows\System32\notepad.exe C:\Windows\Temp\test_out.txt
```

**Safety:** All commands either produce no output (encoded PowerShell prints to console only), error harmlessly (rundll32/mshta execute in a sandboxed HTML application context with no network access), or perform a benign file operation (certutil decodes an already-present system binary). No network connections, no payloads, no persistence.

## Detection logic

### Splunk SPL

```spl
index=winlogs sourcetype="WinEventLog:Security" EventCode=4688
| where NOT match(Account_Name, "(?i)(splunkd|SYSTEM|LOCAL SERVICE|NETWORK SERVICE|DWM-|UMFD-)")
| where NOT match(Account_Name, "\$$")
| eval proc=lower(New_Process_Name)
| eval cmdline=lower(Process_Command_Line)
| eval detection=case(
    match(cmdline, "(-enc|-encodedcommand|-e\s+[a-z0-9+/=]{20,})"),
        "T1059.001 — PowerShell encoded command",
    match(proc, "mshta\.exe") AND match(cmdline, "(vbscript|javascript|http)"),
        "T1218.005 — MSHTA LOLBin",
    match(proc, "rundll32\.exe") AND match(cmdline, "(javascript|vbscript|\.dll,)"),
        "T1218.011 — Rundll32 LOLBin",
    match(proc, "certutil\.exe") AND match(cmdline, "(-decode|-encode|-urlcache|-ping)"),
        "T1105/T1140 — Certutil abuse",
    match(proc, "regsvr32\.exe") AND match(cmdline, "(/s|/u|http|scrobj)"),
        "T1218.010 — Regsvr32 LOLBin",
    match(proc, "(wscript|cscript)\.exe") AND match(cmdline, "\.(js|vbs|wsf|hta)"),
        "T1059.005 — Script interpreter",
    match(proc, "msiexec\.exe") AND match(cmdline, "(http|/q|/quiet)"),
        "T1218.007 — Msiexec remote payload",
    match(proc, "bitsadmin\.exe") AND match(cmdline, "(transfer|download|addfile)"),
        "T1197 — BITSAdmin download",
    true(), null()
  )
| where isnotnull(detection)
| table _time, Account_Name, Creator_Process_Name, New_Process_Name, Process_Command_Line, detection
| sort -_time
```

### How it works

| Step | Purpose |
|---|---|
| `sourcetype="WinEventLog:Security" EventCode=4688` | Scope to Security log process creation events only |
| `NOT match(Account_Name, ...)` — service accounts | Suppress Splunk, SYSTEM, and built-in service account noise |
| `NOT match(Account_Name, "\$$")` | Suppress machine accounts (end with `$`) — computer-initiated process creation |
| `eval proc / cmdline` | Normalise both fields to lowercase for case-insensitive matching |
| `eval detection=case(...)` | Map command line patterns to MITRE technique labels |
| `where isnotnull(detection)` | Surface only events matching a known-malicious pattern — drops everything else |
| `Creator_Process_Name` | Preserved in output — parent process context is critical for triage |

### Why command line pattern matching works here

LOLBin abuse has a defining characteristic: the malicious invocation requires specific arguments that are not used in legitimate contexts. `rundll32.exe` is called legitimately thousands of times per day — but `rundll32.exe javascript:` is not a legitimate invocation pattern outside of deliberate abuse. `certutil.exe -urlcache -split -f http://...` has no benign use case in a corporate environment. Pattern matching on these specific argument combinations produces very high precision with low false positive rates, unlike process-name-only detection which is trivially evaded by binary renaming.

## Dual-source coverage: Event 4688 vs. Sysmon Event 1

Detection #1 in this lab catches T1059.001 encoded PowerShell via Sysmon Event 1. This detection catches the same technique via Security Event 4688. The encoded PowerShell event fired in both — confirming that neither telemetry source is redundant:

| Capability | Sysmon Event 1 | Security Event 4688 |
|---|---|---|
| Process command line | ✅ Yes | ✅ Yes (requires registry key) |
| Parent command line | ✅ Yes | ❌ No |
| Process hash | ✅ Yes | ❌ No |
| User account | ✅ Yes | ✅ Yes |
| Logon ID | ❌ No | ✅ Yes (correlate with 4624) |
| Always available | Requires Sysmon | Built into Windows |
| Config complexity | Medium | Low |

In an environment where Sysmon is unavailable or removed by an attacker, 4688 provides independent detection coverage. In an environment where both are present, Sysmon Event 1 adds hash and parent command line context that 4688 lacks.

## Tuning for production

Event 4688 generates extremely high volume on a busy workstation — 80,000+ events per day in this lab alone, the majority from Splunk's own process activity. The detection query's `isnotnull(detection)` filter reduces this to only labeled matches, but the allowlist requires maintenance:

| Noise source | Mitigation |
|---|---|
| Splunk and monitoring agents | Add to `Account_Name` allowlist by service account name |
| Machine accounts (`DOMAIN\COMPUTER$`) | Already filtered by `\$$` regex |
| Software installers using msiexec | Add known installer hashes or parent processes to allowlist |
| Legitimate certutil operations | Rare in user context — investigate rather than suppress |
| Security tools using rundll32 | Add specific command line patterns to exclusion list |

**Recommended production adjustments:**

- Maintain account allowlist as a Splunk lookup table — easier to update than inline regex
- Add `CommandLine` length filter: very short or very long command lines both warrant attention
- Correlate `Creator_Process_Name` — Office applications spawning any LOLBin is critical-severity regardless of command line content
- Enrich with `4624` logon events using `Logon_ID` to identify the originating session and source IP for remote executions

---

## Limitations

1. **Binary renaming evades process-name matching.** An attacker who copies `rundll32.exe` to `svchost32.exe` and executes it from there will not match `proc` filters. Sysmon Event 1 with hash-based detection mitigates this; 4688 alone does not.
2. **No parent command line.** Unlike Sysmon Event 1, 4688 does not include the parent process command line — only the parent process name. A malicious Word macro spawning rundll32 looks the same as a legitimate one at the 4688 layer without correlating back to the parent's own 4688 event.
3. **Volume is extreme without filtering.** 80,000+ events per day from a single workstation makes retention cost significant at scale. Production deployment requires careful allowlisting and potentially a separate summary index.
4. **Command line logging is a separate opt-in.** If the registry key is missing or removed, 4688 still fires but `Process Command Line` is blank — the entire detection chain fails silently. Monitor for the registry key's presence as a control validation check.
